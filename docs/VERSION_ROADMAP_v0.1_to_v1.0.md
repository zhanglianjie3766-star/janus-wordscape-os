# 雅努斯词境 OS 版本路线图 v0.1 到 v1.0

## 文档定位

本文档记录从当前 `v0.1.0-alpha` 走向 `v1.0` 稳定版的版本计划。

它不是 UI 冻结文档，也不是宪法正文。它是发布节奏、工程质量门槛和验收顺序的路线图。

推荐文件名：

```text
docs/VERSION_ROADMAP_v0.1_to_v1.0.md
```

命名理由：

```text
VERSION_ROADMAP
```

表示这是版本演进路线图。

```text
v0.1_to_v1.0
```

表示它覆盖当前 alpha 到稳定公开版的完整路径。

## 核心判断

当前项目已经达到：

```text
v0.1.0-alpha 可发布
可小范围真实用户试用
具备基础工程审计线
```

但尚未达到：

```text
v1.0 长期稳定版
Anki 级成熟产品
大规模公开用户可信使用
```

如果只是个人使用，当前版本已经可以进入真实使用。

如果要公开发布、让别人下载使用、并逐步形成社区词卡生态，则必须最终走向 v1.0 稳定版。

## 总原则

不要把 v1.0 当成第一次发布的前置条件。

正确节奏是：

```text
先发布 v0.1.0-alpha
找 3 到 5 个真实用户试用
用真实问题倒推 v0.2、v0.3、v0.4
最后进入 v1.0
```

这样既避免过度工程，也不牺牲长期质量。

## v0.1.0-alpha：当前可发布试用版

### 目标

小范围真实用户试用，验证用户是否愿意每天使用。

### 当前能力

- 五个底栏页面已经成型：今日、单词本、统计、图谱、设置。
- 单词本支持二级场景、子页、状态筛选、学习和浏览。
- 今日页只处理待复习队列。
- 点击单词列表词卡视为不认识，写入 `browser_detail + again`，加入今日待复习队列。
- 正式复习评分写入 `daily_task`。
- FSRS 使用 `ts-fsrs`，版本锁定为 `5.4.0`。
- ReviewEvent 记录 `scheduler` 和 `scheduler_version`。
- 区分 `fsrs_raw_state_after` 和产品规则适配后的 `state_after`。
- 支持导入、备份、恢复、数据健康检查。
- 社区词卡包和备份恢复被视为不可信输入。
- 已有 UI 回归脚本、安全 smoke、FSRS golden test。

### 发布门槛

- `corepack pnpm run typecheck`
- `corepack pnpm run build`
- `corepack pnpm run test:fsrs-golden`
- `corepack pnpm run smoke:security`
- `corepack pnpm run smoke:phase11`
- `corepack pnpm audit --prod`

### 验收重点

- 真实用户能否完成导入、学习、复习、备份。
- 手机端是否愿意连续使用。
- 今日待复习、单词本、统计、图谱之间的数据是否一致。
- 用户是否理解“学习中、待复习、已掌握”等状态。

## v0.2.0-alpha：工程结构稳定

### 目标

降低主包维护噪音，让代码结构更经得起审计。

### 必做事项

- 继续拆分 `App.tsx`。
- 将 `DecksView` 移出主文件。
- 拆分单词本子页逻辑。
- 拆分学习卡片、词典式列表、范围设置弹层。
- 拆分图谱渲染、图谱交互、图谱设置、图谱详情弹层。
- 将导入、备份、恢复、学习评分等业务动作收敛到独立 hooks 或 service 模块。

### 验收标准

- `App.tsx` 只负责应用壳、路由和页面组合。
- 业务逻辑模块有清晰边界。
- 组件职责单一。
- 不改变当前 5 个页面 UI 效果。
- 既有 smoke 全部通过。

## v0.3.0-alpha：测试体系补齐

### 目标

把核心规则从“靠人工记忆”升级为“测试可守住”。

### 必做事项

- 持久化写入队列单元测试。
- 备份恢复校验单元测试。
- FSRS adapter 单元测试。
- `browser_detail + again` 产品规则测试。
- 今日待复习队列测试。
- 单词本状态筛选测试。
- 图谱交互状态机测试。
- 导入大小、URL 协议、音频 URL 安全边界测试。

### 验收标准

- 核心数据规则可以脱离浏览器 UI 独立测试。
- smoke 用于端到端验证，unit test 用于守住细粒度规则。
- 任意人修改调度、恢复、队列、图谱交互时，测试能及时暴露回归。

## v0.4.0-beta：数据迁移与兼容稳定

### 目标

让旧数据、旧备份、不同浏览器和不同手机设备都能稳定运行。

### 必做事项

- 建立 `app_version` 与 `backup_schema_version` 的迁移策略。
- 为旧备份建立 migration pipeline。
- 增加 schema migration 测试。
- 增加恢复失败的清晰错误提示。
- 增加真实设备矩阵验证。
- 验证 Chrome、Edge、Safari、微信内置浏览器和主流安卓浏览器。
- 验证不同手机屏幕宽度、不同 DPR、不同安全区。

### 验收标准

- 旧版本备份可以恢复。
- schema 升级不会破坏真实学习数据。
- 手机端布局在主流设备上不错行、不遮挡、不丢交互。
- IndexedDB、localStorage、Service Worker 边界清晰。

## v1.0：稳定公开版

### 目标

成为可以面向公开用户长期使用的稳定版本。

### 必做事项

- CI 全绿。
- Release 流程固化。
- GitHub Actions 自动构建、测试、打包。
- GitHub Release 附带 `dist.zip`、样板词卡包、快速使用说明。
- LICENSE、NOTICE、PRIVACY、SECURITY、CHANGELOG 稳定。
- FSRS / ts-fsrs 版权声明和算法说明完整。
- 备份、恢复、迁移经过多轮真实用户验证。
- 真实设备矩阵通过。
- 用户手册稳定。
- 开源贡献边界、词卡包社区规则和品牌授权说明稳定。

### v1.0 发布门槛

- 真实用户试用反馈已闭环。
- 无已知数据丢失问题。
- 无已知备份恢复阻断问题。
- 无已知算法审计缺口。
- 无高危依赖漏洞。
- 无跨端严重 UI 兼容问题。
- 项目文档足够让新开发者读懂系统边界。

## 长期质量底线

只要项目进入公开发布和社区生态，以下事项必须长期保持：

- 数据不能轻易丢。
- 备份必须可靠恢复。
- 算法结果必须可审计。
- schema 升级必须可迁移。
- 不可信词卡包不能伤害用户。
- 手机和浏览器兼容必须稳定。
- 发布流程必须可重复、可回滚。
- 第三方依赖必须版本锁定并保留版权声明。

## 不做事项

以下事项不作为 v0.1.0-alpha 的前置条件：

- 完整云同步。
- 用户账号体系。
- 大规模社区词卡市场。
- 复杂权限系统。
- 商业支付系统。
- 多语言全量本地化。

这些可以进入 v1.x 或更后续阶段。

## 下一步建议

当前最合理的下一步是：

```text
发布 v0.1.0-alpha
找 3 到 5 个真实用户试用
记录真实问题
按问题优先级进入 v0.2.0-alpha
```

不要为了追求一次性完美而推迟真实使用。雅努斯词境 OS 的长期质量应该来自稳定基线、真实反馈和持续审计，而不是发布前的无限设计。
